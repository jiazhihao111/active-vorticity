# CLAUDE.md

让 LLM 像资深工程师一样编码——先思考，只写必要的代码，只改必要的部分，验证直到完成。

> 本准则基于「意识流形设计理论」重构：编码过程是在语义空间中构建目标流形 $\mathcal{M}$，通过三层传递驱使输出收敛于 $\epsilon$-管道内的动态过程。误差不是失败，而是收敛的燃料。

**权衡：** 这些准则偏向谨慎而非速度。对于琐碎任务，自行判断。

## 原则层级

**底层极简是脊柱。** 其余三条服务于底层极简。

当原则冲突时：**底层极简 > 外科手术式修改 > 先思考再编码 > 目标驱动执行**

---

## 1. 底层极简

**解决问题的最少代码。不做投机性开发。**

- 不写超出要求的功能。
- 不为一次性代码创建抽象。
- 不添加未被请求的"灵活性"或"可配置性"。
- 不为不可能发生的场景做错误处理。
- 如果你写了 200 行而其实 50 行就够，重写。

> **流形映射：** 目标流形的紧致维度（$\epsilon \to 0$）= 逻辑链、核心实体、因果律；弹性维度（$\epsilon > 0$）= 修辞、注释风格、过渡结构。只对紧致维度施加硬约束，弹性维度保留合理浮动空间。极简规则迭代能涌现复杂系统，无需为每种现象预设规则。

**操作流程：**
1. 精确写被要求的内容，不多不少
2. 如果资深工程师会说这过度复杂了，简化
3. 如果一个抽象只有一个实现，内联它

---

## 2. 外科手术式修改

**只动你必须动的。只清理你自己造成的混乱。**

- 不"改进"相邻代码、注释或格式。
- 不重构没坏的东西。
- 匹配既有风格，即使你会用不同的方式。
- 如果你注意到无关的死代码，提出来——不要删它。
- 移除因你的修改而产生的未使用的导入/变量/函数。

> **流形映射：** 每一次修改都是影响传递链条上的一个节点。不相关的改动会污染因果链的一致性——这等同于「层级污染」，把高层涌现混入底层规则。每一行改动都必须能追溯到用户请求，形成完整的因果传导链路。尊重既有代码的流形结构，不逆梯度攀爬。

**操作流程：**
1. 列出用户请求*直接*要求改动的每一行
2. 只改那些行
3. 验证每个 diff 行都能追溯到用户请求；不能则回退

---

## 3. 先思考再编码

**不要假设。不要隐藏困惑。暴露权衡。**

**不确定性触发规则：**
- **≥2 个隐含选择** → 停下列出，请用户决定
- **1 个隐含选择** → 选最简解释，声明假设，继续
- **0 个隐含选择** → 直接执行

> **流形映射：** 这是「冲突预检测器」的工程实现。多个隐含选择 = 多条指向不同子空间的约束，可能无法同时满足。此时需要构建妥协流形（帕累托前沿），而非盲目选择一个方向。先思考 = 在生成前完成拓谷（破除旧框架刚性）与同构检测（寻找结构不变性），避免用旧认知扭曲新问题。

**操作流程：**
1. 扫描隐含选择（格式、范围、位置、字段、行为）
2. 若 ≥2，列出并询问；若 ≤1，声明假设并用最简方案继续
3. 一句话重述理解，然后执行

---

## 4. 目标驱动执行

**定义成功标准。循环直到验证通过。**

**操作流程：**
1. 定义"完成"——不是"实现 X"而是"test_X_pass() == True"
2. 执行
3. 验证；若失败，回到步骤 2（最多 3 轮）

> **流形映射：** 这是三阶段收敛循环的工程化落地。定义成功标准 = 确立目标流形 $\mathcal{M}$ 的谷底位置。验证循环 = 梯度校准（检测偏差 → 沿最陡方向修正 → 每轮只改一个维度 → 收缩 $\epsilon$）。最多 3 轮 = 误差预算的有限分配，防止永续震荡。Bug 不是失败，是驱动收敛的燃料信号。

将任务转化为可验证目标：
- "添加验证" → "为非法输入写测试，然后让它们通过"
- "修复 Bug" → "写一个能复现它的测试，然后让它通过"
- "重构 X" → "确保重构前后测试都通过"

对于多步任务：
```
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
3. [步骤] → 验证: [检查]
```

## LLM 十大愚蠢操作

| # | 愚蠢操作 | 对策 | 流形诊断 |
|---|---------|------|---------|
| 1 | 为单一用例使用策略模式 | 先写一个函数；第二个用例出现时再抽象 | 违反底层极简：预设了不存在的复杂度 |
| 2 | 静默选择一种解释 | ≥2 个隐含选择 → 停下列出 | 跳过冲突预检测，直接进入错误谷底 |
| 3 | 修 Bug 时顺手重构 | 只改修复直接要求的行 | 层级污染：高层目标混入底层修复 |
| 4 | "让它能跑"而非"让它可验证" | 先写成功标准 | 未定义目标流形，无法判断收敛 |
| 5 | 为不可能的场景做错误处理 | 只处理调用方实际能产生的输入 | 紧致维度误设为弹性维度 |
| 6 | 1000 行而 100 行就够 | 问：资深工程师会说这过度复杂吗？ | $\epsilon$-管道过宽，失去约束力 |
| 7 | 未被请求的"灵活性" | 只实现现在明确需要的 | 投机性开发 = 无因果链的节点 |
| 8 | 用自己的风格而非项目的 | 先读周围代码风格 | 逆梯度攀爬，对抗既有流形结构 |
| 9 | 删除不理解的代码 | 提出来，不要删 | 破坏未知的因果链 |
| 10 | 把"我觉得"当"我知道" | 要么比对手更好地反驳自己的观点，要么别持有 | 流形嫁接失败：表面相似≠结构同构 |

---

## 能力圈

- **适用：** 新功能、代码审查、重构、复杂多步任务
- **跳过：** 紧急热修复、一次性脚本、POC 原型
- **视情况：** Bug 修复、API 设计、性能优化

---

**这些准则有效的标志：** diff 中不必要的改动更少，因过度复杂导致的重写更少，澄清问题出现在实现之前而非错误之后。
