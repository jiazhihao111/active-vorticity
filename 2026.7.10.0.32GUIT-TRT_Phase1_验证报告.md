# GUIT-TRT Phase 1 验证报告：Qwen2.5-7B 显存压缩与热力学指标

**日期**：2026-07-10
**模型**：Qwen2.5-7B-Instruct (bf16, 28层, 3584维)
**GPU**：NVIDIA RTX 5080 Laptop (16GB VRAM)

---

## 一、实验设计

### 测试场景
- **输入**：中文因果叙事（~150 tokens），含明确的因果链（下雨→取消爬山→买牛奶面包→路被封→绕道后门）
- **问题**：小明最终是如何回家的？他买了什么？
- **正确答案**：绕道从后门回家；一盒牛奶和两个面包

### 实验条件
| 条件 | 说明 |
|------|------|
| Baseline | 原生 HF `model.generate()`，无监控 |
| Probe | 逐token生成 + ThermoProbe监控，无淘汰 |
| KV-128 | CausalKV淘汰，max_capacity=128 |
| KV-64 | CausalKV淘汰，max_capacity=64（极限压缩） |

### 关键参数
- α* = 1.41 (Qwen2.5-7B校准值)
- γ = 0.01
- sink_tokens = 4, recent_tokens = 10
- max_new_tokens = 80

|------|-----|
| P_c/P_raw 均值 | **0.1207** |
| P_c/P_raw 标准差 | 0.1029 |
| P_c/P_raw 范围 | [0.0006, 0.5473] |
| Causal impact 均值 | 222.73 |
| Causal impact 标准差 | 60.86 |

**关键发现**：
- P_c/P_raw 均值0.12远高于v16实验的pos值(0.0013)，原因是**逐token生成模式**下模型需要处理chat template等非因果内容
- 范围[0.0006, 0.5473]跨度近3个数量级，存在明确的"因果滑行区"与"因果转折区"的区分
- **阈值τ建议**：0.05-0.10区间（覆盖大部分平稳区，仅拦截极端偏离）

## 四、新发现与红队反诘

### 🔴 新问题1：Probe模式的"观察者效应"
- **现象**：Probe模式（逐token生成+output_hidden_states=True）的生成结果与Baseline**完全不同**
- **根因**：`output_hidden_states=True`改变了HF内部的计算图优化路径，且逐token手动解码与`model.generate()`的采样策略不同
- **影响**：这不是GUIT理论的缺陷，而是**验证方法论的缺陷**——不能在Probe模式下评估生成质量
- **修正**：Phase 2必须使用Hook方式注入（不改变generate流程），而非手动逐token解码

### 🔴 新问题2：短文本下VRAM节省不显著
- **现象**：所有条件下Peak VRAM几乎相同（12.92-12.99 GB）
- **根因**：短文本（~230 tokens）的KV Cache仅占~0.07GB，淘汰节省的量被`output_hidden_states`的额外开销完全掩盖
- **修正**：必须在**长文本（4k-32k tokens）** 下测试，KV Cache才会成为VRAM瓶颈

### 🟡 新问题3：KV-64的幻觉比预期严重
- **现象**：max_capacity=64时淘汰16个token，生成内容完全崩溃
- **根因**：输入prompt本身~150 tokens，KV-64容量不足以保留完整上下文
- **修正**：max_capacity必须 > input_length + recent_tokens + sink_tokens

## 五、结论与下一步

### 核心结论
1. **P_c/P_raw指标可提取且分布合理**：range=[0.0006, 0.5473]，存在可区分的"因果滑行区"与"转折区"
2. **CausalKV淘汰逻辑可运行**：KV-64成功淘汰16个token
3. **短文本VRAM节省不显著**：需在长文本(4k+)下验证
4. **Probe模式存在观察者效应**：需改用Hook注入方式

### 下一步行动
1. **改用Hook注入**：通过`register_forward_hook`提取隐状态，不改变`model.generate()`流程
2. **长文本测试**：构造4k+ token的输入，测量VRAM节省
3. **阈值标定**：在Hook模式下标定τ的最佳值
4. **Early Exit验证**：基于Hook的P_c/P_raw实时计算，测试浅层退出

### 固化度更新
- "降低显存提高运行速度"方案的GUIT固化度：**0.45**（弱固化/需更多实证）
- 较初始评估(0.30)提升：区分信息已补全，但复制信息仍不足（需长文本+Hook验证）
